# План: Эмуляция GSM-модема в RFC-2217 прокси

## Контекст

Опросное ПО (Windows) умеет подключаться к счётчикам через GSM-CSD — отправляет стандартные модемные AT-команды (`ATD<номер>`) через COM-порт. У ПО нет режима кастомных AT-команд, но есть виртуальный COM с поддержкой RFC-2217.

**Цель:** научить прокси эмулировать GSM-модем, чтобы `ATD<номер>` работал как `AT+CONNECT=<номер>`.

```
Windows ПО (GSM-CSD)
    → ATD<device_id>
    → Virtual COM (RFC-2217)
    → TCP
    → RFC-2217 Proxy (эмуляция модема)
    → Device (за NAT)
```

## Этап 1: Диагностика — что отправляет клиент

Прежде чем писать эмуляцию, нужно точно знать какие AT-команды и в какой последовательности отправляет Windows-программа в режиме GSM-CSD.

### 1.1 Добавить логирование всех входящих данных

В `internal/connection/handler.go` — логировать **каждый байт** до распознавания AT-команды:

- Все данные до AT-команды (RFC-2217 negotiations, мусор, etc.)
- Саму AT-строку целиком (до `\r\n`)
- Нераспознанные команды — логировать, но **не отключать** соединение

### 1.2 Временный режим "AT sniffer"

Вместо `ERROR` на неизвестные команды — отвечать `OK` и продолжать слушать. Это позволит записать **полную** последовательность инициализации модема.

Ожидаемая последовательность (типичная для GSM-CSD):
```
AT              → OK
ATE0            → OK (отключить эхо)
AT+CGMI         → OK или имя производителя
AT+CPIN?        → +CPIN: READY
ATD<номер>      → CONNECT 9600
... данные ...
+++             → OK
ATH             → NO CARRIER
```

### 1.3 Тестирование

1. Запустить прокси с `DEBUG=true`
2. Настроить виртуальный COM на Windows → подключить к прокси
3. Запустить опросное ПО в режиме GSM-CSD
4. Собрать лог всех AT-команд

**Результат этапа:** точный список AT-команд от клиентского ПО.

## Этап 2: Эмуляция GSM-модема

После получения реального лога — реализовать обработку команд.

### 2.1 Новый файл: `internal/connection/modem.go`

Модемный обработчик AT-команд со словарём ответов:

| Команда | Ответ |
|---------|-------|
| `AT` | `OK` |
| `ATE0` / `ATE1` | `OK` |
| `ATZ` | `OK` |
| `ATH` / `ATH0` | `OK` или `NO CARRIER` |
| `AT+CGMI` | `proxy-rfc2217` |
| `AT+CPIN?` | `+CPIN: READY` |
| `AT+CSQ` | `+CSQ: 31,0` |
| `ATI` | `RFC2217 GSM Emulator` |
| `ATD<номер>` | → `AT+CONNECT=<номер>` → `CONNECT 9600` |
| `ATDT<номер>` | аналогично ATD |
| `ATDP<номер>` | аналогично ATD |
| `+++` | переход в командный режим, `OK` |

### 2.2 Изменение в `internal/connection/handler.go`

В основном цикле `Handle()`:

```
case CmdReg:     → handleDevice()      // как сейчас
case CmdConnect: → handleClient()      // как сейчас
case CmdDT, CmdDP:
    if cmd.Param != "" {
        // ATD<номер> — это "звонок", обработать как AT+CONNECT
        handleClient(ctx, conn, reader, cmd, remoteAddr)
        return
    }
    // ATDT без параметра — ответить OK, ждать дальше
    WriteOK(conn)
    continue
default:
    // Неизвестная AT-команда — попробовать как модемную
    if handleModemCommand(conn, cmd) {
        continue  // Отвечено OK, ждём следующую
    }
    WriteError(conn)
    return
```

### 2.3 Изменение ответа для GSM-CSD клиента

При подключении через `ATD<номер>`:
- Ответ: `CONNECT 9600\r\n` (вместо `OK\r\n`)
- Опросное ПО ожидает именно `CONNECT <speed>`

При разрыве сессии:
- Отправить `NO CARRIER\r\n` клиенту

### 2.4 Обработка `+++` (escape sequence)

В bridge (`internal/session/bridge.go`):
- Детектировать `+++` с guard time (1 сек тишины до и после)
- Переключить клиента в командный режим
- Ждать `ATH` → завершить сессию

## Этап 3: Расширение парсера AT-команд

### 3.1 `internal/connection/protocol.go`

Расширить парсер для распознавания произвольных AT-команд:
- `AT` (без параметров)
- `ATE0`, `ATE1`, `ATZ`, `ATH`, `ATI`
- `AT+CGMI`, `AT+CPIN?`, `AT+CSQ`
- Любые `AT...` — как generic modem command
